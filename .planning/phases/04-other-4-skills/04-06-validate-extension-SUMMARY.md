# Phase 04 Plan 06: validate-extension Summary

CI-side audit harness for the four Phase 4 skill bundles — `auditPhase4Skills()` enforces frontmatter compliance (allowed-tools present, /zama-deploy carries `disable-model-invocation: true`), required-asset manifest (templates + scripts), sync-marker presence (one `@sync:prompt:closing-summary-<skill>` per bundle), deprecated-import grep, and the DEPLOY-03 invariant that no Sepolia hex address may live in deploy/ source outside test fixtures.

## What changed

### scripts/validate.ts
- New exported function `auditPhase4Skills(rootDir)` returning `{ ok, errors[] }`.
- New `PHASE4_MANIFEST` with one entry per skill (`contract`, `test`, `deploy`, `frontend`); declares `skillMd`, `scripts[]`, `templates[]`, `requiresDisableModelInvocation`, `syncMarkers[]`.
- Helper `loadDeprecationBanlist()` reads `plugins/zama-skills/shared/deprecated-imports.json` and returns the keys of the `deprecated` section (intentionally NOT `incompatible` — see Decisions).
- Helper `findDeprecationHits()` matches deprecated identifiers only inside import-shape contexts (`from 'X'`, `require('X')`, `import 'X'`, JSON `"X":` key); skips comment lines (reuses `isCommentLine` from the existing init-asset audit).
- Helper `walkBundleFiles()` reuses the existing `walkFiles` recursive walker scoped to each skill bundle.
- Hex-address grep: `\b0x[0-9a-fA-F]{40}\b` per line, deploy-only, allowlisted paths matching `(?:__fixtures__|\.test\.[tj]sx?$|\.test\.sol$)`.
- Wired into `runValidate()` immediately after the init asset audit; failure prints red errors and exits 1.

### scripts/validate.test.ts
Added `describe('auditPhase4Skills', ...)` block with 9 cases:
1. Happy path — real repo passes with zero errors.
2. Missing template (`contract.sol.tpl` removed) → flagged with skill + filename.
3. Missing script (`sepolia-addresses.ts` removed) → flagged with skill + filename.
4. Deploy `disable-model-invocation: false` → flagged.
5. Missing `allowed-tools` from contract frontmatter → flagged.
6. `import "fhevmjs"` injected into `frontend/.../fhe.ts.tpl` → flagged with skill + filename.
7. Hex address pinned in `deploy/SKILL.md` → flagged (DEPLOY-03).
8. Hex address inside `deploy/scripts/__fixtures__/sample.json` → NOT flagged (allowlist works).
9. Stripping `<!-- @sync:prompt:closing-summary-deploy -->` opener → flagged.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Critical functionality] Wire missing closing-summary sync markers in 3 SKILL.md files**
- **Found during:** Task 1 (audit on real repo flagged 4 missing markers)
- **Issue:** Wave 1 plan 04-05 created per-skill closing-summary fragments under `plugins/zama-skills/shared/prompts/closing-summary-{contract,test,deploy,frontend}.md`, but the contract/test/frontend SKILL.md files never gained their transclusion markers, and deploy used the generic `@sync:prompt:closing-summary` marker (pointing at the obsolete generic fragment).
- **Fix:** Appended `## Closing Summary` section + `<!-- @sync:prompt:closing-summary-<skill> -->\n<placeholder>\n<!-- @endsync -->` to contract/test/frontend SKILL.md; renamed deploy's marker to `closing-summary-deploy`. Ran `pnpm sync` to populate bodies (also regenerated generic mirrors).
- **Files modified:** plugins/zama-skills/skills/{contract,test,deploy,frontend}/SKILL.md, generic/{contract,test,deploy,frontend}.md
- **Commit:** 713d063
- **Why this matters:** The plan's audit invariant (`syncMarkers: ['@sync:prompt:closing-summary-<skill>']`) is non-negotiable. Without this fix, the audit would correctly flag pre-existing drift, but `pnpm validate` would have exited 1 on the live repo — failing the plan's success criterion ("`pnpm validate` exits 0 on the real repo after Plans 01-05").

**2. [Rule 1 — Bug] Restrict deprecation-grep banlist to `deprecated` section only**
- **Found during:** Task 1 (initial implementation included `incompatible` keys → 4 false positives flagging legitimate `from "hardhat"` imports in deploy/test templates)
- **Issue:** Stripping `@^3` from `hardhat@^3` to make a substring match flagged any `hardhat` import — but Hardhat 2.x is the *required* runtime; only v3 is incompatible. The grep can't distinguish at the import-string level.
- **Fix:** `loadDeprecationBanlist()` now returns only `Object.keys(parsed.deprecated)` (i.e. `fhevmjs`, `fhevm`). Version-bound incompatibility is a package.json install-time concern (already enforced via `pinned-versions.json` checks in `scripts/lib/versions.ts`), not an import-string check.
